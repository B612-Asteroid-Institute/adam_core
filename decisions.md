- 2026-04-17 08:49 EDT: SPICE parity fixture uses a two-layer capture (raw CSPICE km/km-s + adam-core AU/AU-d API layer) and bit-exact tolerance at the raw layer. Reason: Rust FFI will call CSPICE with identical inputs and must reproduce bit-for-bit; any discrepancy indicates a kernel mismatch or boundary bug, not "close enough" floating-point drift. - 2026-04-18 08:15 EDT: Rust-side CSPICE is a distinct toolkit instance (its own kernel pool, error stack, SPK cache) from spiceypy's. Reason: linking both against the same dylib is not feasible when each toolchain statically links its own CSPICE .a; on Apple Silicon the `downloadcspice` feature even fetches the wrong architecture. Consequence: callers must double-load kernels (once via spiceypy, once via `spice_furnsh`) during the transition; a small perf cost at startup but avoids single-instance coupling complications. Plan: add a `setup_SPICE` hook that fans kernel paths to both instances.
- 2026-04-18 08:15 EDT: Rust-vs-spiceypy CSPICE parity asserts at science-grade tolerance (10mm pos / 500 nm/s vel / 1 ns lt / 1e-13 rot), not bit-exact. Reason: two independent CSPICE compilations reorder FP ops at machine epsilon; demanding bit-exactness would block adoption for no science reason (10mm is already below any realistic positional error budget).
- 2026-04-18 08:15 EDT: Vendor the correct arm64 CSPICE under `vendor/cspice/` and point `CSPICE_DIR` at it through `.cargo/config.toml` `[env]`. Reason: the `cspice-sys` `downloadcspice` feature checks `target_arch="arm"` when it should be `"aarch64"` and ships the x86_64 build on Apple Silicon; vendoring is the most robust fix without patching the upstream crate.

- 2026-04-18 08:31 EDT: `get_perturber_state` routes J2000 and ECLIPJ2000 queries through the pure-Rust `NaifSpk` reader (not the CSPICE FFI bridge). Reason: the pure-Rust reader is a dependency-free, pip-installable path that doesn't require vendoring CSPICE archives or per-platform toolkit builds — it evaluates DE440 Chebyshev directly and is already 4-10x faster per call. Fallback to spiceypy is kept for (a) ITRF93 (time-varying, needs PCK/EOP — Phase H), (b) bodies outside DE440 (e.g., asteroids in user-supplied kernels), and (c) out-of-coverage ET. Rationale for fallback over fail-loudly: production callers pass in OriginCodes that span DE440 planets + user-furnished small bodies, and the kernel pool is a runtime input; failing loudly here would break existing workflows that work today under spiceypy.
- 2026-04-18 08:31 EDT: Parity tolerance philosophy for pure-Rust-backed IOD vs legacy IOD: when algorithms are numerically ill-conditioned (Gibbs on closely-spaced triplets amplifies ~1e8x), the test tolerance should be set to a physically meaningful threshold (science-grade), not to machine epsilon. Machine-epsilon comparisons between rust and legacy paths are fragile and depend on FP ordering coincidences that break under 1-ULP input perturbations. Continue to enforce 1e-12 tolerance where algorithms are well-conditioned (straight-line Gibbs test on synthetic inputs), and loosen only where a specific instability is understood and documented.

- 2026-04-19 17:48 EDT: Prefer input-space fixes over tolerance bumps when parity tests fail. Reason: a tolerance bump hides the amplification factor driving the failure and drifts the test away from proving algorithmic equivalence; fixing the sampler to pick a well-conditioned triplet both (a) tightens the parity assertion (1e-12 vs 1e-6) and (b) makes the test actually validate what it claims to validate. Revises the 2026-04-18 08:31 tolerance decision for the specific Gauss IOD case — that test now runs at bit-parity (atol=1e-12). The general "science-grade tolerance where algorithm is ill-conditioned" philosophy stands, but in the Gauss IOD case the ill-conditioning came from the sample selection, not an intrinsic algorithm property.
