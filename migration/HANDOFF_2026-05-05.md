## Executive summary

The migration goal (API-majority Rust-backed `adam-core` while preserving the
public Python API) is unchanged. Since `661046ab` (last commit referenced in
HANDOFF_2026-04-30):

1. The single-thread warm-speed gate floor was raised from 1.0x to **1.2x
   p50/p95** with no active waivers (commit `b67d9126`).
2. Three waves of macOS-specific fast paths were added under
   `rust/adam_core_rs_coords/src/`:
   - Wave 1 (`3551387b` … `6cf128dc`): vForce/NEON tile pipeline for the four
     photometry kernels, with thread-local scratch and per-kernel tile sizes
     `PHOT_VFORCE_TILE_ROWS = 8192`, `PHOT_VFORCE_PHASE_TILE_ROWS = 4096`.
   - Wave 2 (`a3780470`): pack `phi1`/`phi2` into a single 2N scratch for one
     fused `vvexp`.
   - Wave 3 (`fd182ebd`, `b58bebb6`): macOS batched-acos pipeline for
     `cartesian_to_keplerian_flat6` and `cartesian_to_cometary_flat6` via a
     shared `c2k_macos.rs` helper module.
3. macOS fast-path thresholds lowered to `C2K_MACOS_MIN_ROWS = 64` and
   `PHOT_VFORCE_MIN_ROWS = 64` so the canonical `parity_fuzz --n 128` actually
   exercises them (commit `e94a35bc`).
4. The Apple-specific tile loops were originally serial `for ..` loops, which
   accidentally capped production native execution at one core for the
   vForce work. Commit `657d005c` replaced them with
   `out.par_chunks_mut(TILE).enumerate().for_each(...)`, leveraging the
   existing thread-local scratch so production native execution gets ~2.18x –
   ~2.87x scaling on Apple Silicon.
5. The harness thread-mode token was renamed `native` → `multi-thread`
   (commit `6faac94d`) to remove confusion with `_rust_native` and other uses
   of `native`. `native` remains accepted as a deprecated backward-compat
   alias.

The single-thread gate currently shows: **all 23 small-n APIs PASS**, **21/23
large-n PASS**, with two structural large-n misses on
`dynamics.generate_ephemeris_2body` (~1.05x) and
`photometry.predict_magnitudes` (~0.96x — JAX/XLA fusion advantage on the
specific bandpass-table read-modify-write).

Recent decisions added in this wave (now in `decisions.md`):

- The single-thread warm-speed gate floor is **1.2x** at both p50 and p95
  for every lane/API. There are no active lane-scoped waivers.
- Single-thread is the **gate methodology**: comparing multi-thread Rust
  vs multi-thread JAX is hard because each library's threading layer
  scales differently and this would obscure port-quality regressions.
  The single-thread mode caps `RAYON_NUM_THREADS`, `OMP_NUM_THREADS`,
  `OPENBLAS_NUM_THREADS`, `MKL_NUM_THREADS`, `VECLIB_MAXIMUM_THREADS`,
  `NUMEXPR_NUM_THREADS`, `JAX_NUM_THREADS`, and `XLA_FLAGS` to 1 on BOTH
  Rust and the legacy subprocess.
- Multi-thread mode is the **production-realism mode**: it removes the
  caps on BOTH sides so Rust Rayon AND legacy NumPy/JAX/BLAS can scale
  across available cores. It is required as labeled scaling evidence,
  not as a gate.
- Apple-specific fast paths (vForce/NEON) **must retain Rayon
  parallelism**. Single-thread improvements that serialize a previously-
  parallel path are a regression. (See review finding 3 below for the
  specific incident this rule was authored from.)
- Endpoint parity for photometry near 0°/180° must preserve baseline
  rounding behavior; `ACOS_ENDPOINT_GUARD = 1 - 1e-9` is the chosen
  guard band.
- `photometry.predict_magnitudes` 0.96x miss is structural (JAX/XLA
  fuses the bandpass-table read-modify-write that we cannot fuse in
  the same way without a polynomial-acos rewrite).
- Cold timing remains multi-thread by default (one-shot production
  realism). Cold cache is currently empty.
- Use macOS/aarch64 NEON + Apple Accelerate vForce directly (no new
  SIMD dependency).
- Workload shapes preserve axes (`orbits × epochs`, `orbits × observers`,
  etc.) so the `--shape` lane can drive both Rust and legacy with the
  same shaped inputs.
- Do not commit `decisions.md` / `journal.md`; they are local-only
  grounding files.

### 3. Rayon re-parallelization (commit `657d005c`)

Important correctness fix called out by the user during review.

The wave 1–3 macOS code walked tiles via `for row_offset in (0..n).step_by(...)`,
which is sequential. That accidentally capped production native execution at
one CPU core for the vForce work and was a regression vs the cross-platform
scalar path which already used Rayon `par_chunks_mut`.

Commit `657d005c` replaced the sequential tile walks in all six macOS
entry points with `out.par_chunks_mut(TILE).enumerate().for_each(...)`,
where each Rayon worker enters the existing thread-local scratch
(`VFORCE_SCRATCH` / `SCRATCH`). Single-thread gate timings unchanged
within noise; production multi-thread now scales:

| API                                                          | Single | Multi-thread | Multi/Single |
|---|---|---|---|
| photometry.calculate_phase_angle (n=50000)                   | 197 µs | **72 µs**    | **2.74x**    |
| photometry.calculate_apparent_magnitude_v (n=50000)          | 571 µs | **204 µs**   | **2.80x**    |
| photometry.calculate_apparent_magnitude_v_and_phase_angle    | 678 µs | **236 µs**   | **2.87x**    |
| photometry.predict_magnitudes (n=50000)                      | 602 µs | **255 µs**   | **2.36x**    |
| coordinates.cartesian_to_keplerian (n=20000)                 | 2554 µs | **1174 µs** | **2.18x**    |
| coordinates.cartesian_to_cometary (n=20000)                  | 2366 µs | **1084 µs** | **2.18x**    |

In production multi-thread, all six APIs beat JAX legacy by 2.15x – 3.72x,
including `predict_magnitudes` which fails the single-thread gate at
0.96x but delivers 2.15x in production where actual throughput matters.

### 4. Thread-mode rename (commit `6faac94d`)

User direction: renamed the harness thread-mode token `native` →
`multi-thread`. `native` was overloaded with `_rust_native`, native
binary code, and native-OS threads.

Canonical token is now `multi-thread`. `native` is accepted everywhere
as a deprecated backward-compatibility alias and is normalized to
`multi-thread` by `validate_thread_mode` before any cache key, env
apply, or artifact decision is taken. Existing scripts/CI invoking
`--threads native` keep working without change.

Files: `migration/parity/_threading.py` (ThreadMode literal, alias map,
docstring); `migration/parity/parity_speed.py` (SpeedResult default,
measure/measure_lanes signatures, `_result_mode` fallbacks, argparse
choices); `migration/parity/parity_main.py` (argparse choices);
`migration/scripts/rust_backend_benchmark_gate.py` (argparse choices);
`pyproject.toml` (`rust-parity-speed-multithread` script);
`migration/benchmark_governance.md` (wording).

Tests: `src/adam_core/tests/test_rust_migration_status.py` extends
`test_latency_gate_defaults_to_single_thread_policy` with a
`multi-thread` token and adds two new tests:

- `test_thread_mode_native_is_deprecated_alias_for_multi_thread`
- `test_multi_thread_mode_removes_caps_for_both_rust_and_legacy`

The latter is load-bearing: it locks the contract that BOTH Rust
(Rayon) and the legacy baseline (NumPy/JAX/XLA/BLAS) get caps removed
in `multi-thread` mode, so apples-to-apples production-scaling
comparisons are possible.

16/16 tests in `test_rust_migration_status.py` pass.

Empirical note from a `multi-thread` legacy bench against
`coordinates.cartesian_to_keplerian` n=20000:

- single        = 3394 µs
- multi-thread  = 3748 µs
- native alias  = 3431 µs (matches multi-thread, not single)

The env IS uncapped on the legacy subprocess (alias matches non-alias),
but JAX/XLA `vmap(cartesian_to_keplerian)` does not scale on this
workload size — XLA serializes intra-op work. Multi-thread production
headroom for c2k is therefore captured ENTIRELY by Rust Rayon (~2.18x),
giving Rust a ~3x absolute production-multi-thread advantage over
legacy on the same workload. This is exactly the asymmetry the user
flagged: single-thread is the fair gate, multi-thread is where Rust
delivers the real production win.

## Pending direction calls

The next agent should pick one path in each of these two questions
before resuming substantive code work.

### Q1. Platform direction for the macOS-specific code

- **P1**: Replace the macOS-specific NEON+vForce paths with a
  cross-platform SIMD layer (e.g. via `pulp` or hand-written
  scalar/SSE2/AVX2/NEON behind a trait). Pro: portable speedup. Con:
  new dependency, new failure modes, must re-validate.
- **P2**: Keep macOS NEON+vForce but also write per-platform
  AVX2/AVX-512 fast paths for x86_64 Linux. Pro: maximum perf
  everywhere. Con: most code volume; high maintenance.
- **P3**: Revert the Apple-specific paths and keep only the
  cross-platform algorithmic improvements (Rayon, fused geometry
  pipeline, packed scratch). Pro: simplest tree, no `target_os`
  conditionals. Con: gives up ~1.5x – 2x of the wave 1–3 macOS gains.
- **P4**: Document `target_os = "macos", target_arch = "aarch64"`
  as a platform-conditional gate and accept that the warm-speed gate
  thresholds may differ across CI hosts. Pro: keeps current code as-is.
  Con: governance burden.

The user has not yet picked a path. Sub-agent recommendation in the
2026-05-04 review pass was P1 (or P3 if P1 is too costly).

### Q2. Last-mile single-thread misses

- **A**: `dynamics.generate_ephemeris_2body` ~1.05x. Plan:
  shape-aware orbit×epoch decomposition. Each test workload is 400
  orbits × 50 epochs; computing `OrbitConstants` once per orbit
  saves 49/50 of the per-row orbit work, and warm-starting the
  chi/lt iteration across epochs saves ~50% of the Kepler-iteration
  time per row. Estimated path to 1.4x – 1.6x.
- **B**: `photometry.predict_magnitudes` ~0.96x. Plan: replace the
  `acos` round-trip with a polynomial-acos that fuses with the
  bandpass-table read-modify-write, mirroring the JAX fusion. Risk:
  endpoint parity near 0° / 180° (`ACOS_ENDPOINT_GUARD` already
  applies). Estimated path to 1.2x – 1.4x; structural otherwise.
- **C**: Stop and regenerate canonical artifacts with the current
  21/23 large-n PASS rate. Document the two structural misses as
  known and continue to RM-P1-016. Lowest-risk option.

The user has indicated the gate is single-thread but the production
goal is overall throughput. With multi-thread already at 2.15x – 3.72x
for ALL six macOS-path APIs (including `predict_magnitudes`), option
C is defensible from a production-throughput standpoint; A or B are
needed only if every lane must clear the single-thread gate.
